### Problème 1 : Absence d'élément landmark `<main>`

**Critère WCAG :** 1.3.1 – Info et Relations (Niveau A)
**Sévérité :** Critique
**Localisation :** `signin/page.tsx:74` + `app/layout.tsx:28`
**Élément :** `<div className="... min-h-screen bg-white">`

**Problème :**
Toute la page est enveloppée dans des `<div>` génériques. Ni le layout racine ni la page signin ne définissent un élément `<main>`. Sans landmark `<main>`, les utilisateurs de lecteur d'écran ne peuvent pas naviguer directement vers le contenu principal de la page.

**Impact :**
Les utilisateurs aveugles utilisant NVDA, VoiceOver ou JAWS ne peuvent pas utiliser le raccourci "aller au contenu principal" (généralement `H` ou touche de raccourci landmark). Ils doivent traverser tout le contenu séquentiellement.

**Correction :**

```tsx
// Avant — signin/page.tsx:74
<div className={`${plusJakarta.variable} font-sans min-h-screen bg-white`} data-theme="light">
  <div className="flex min-h-screen overflow-hidden">
    {/* Left Panel */}
    <div className="hidden lg:flex lg:w-1/2 ...">
    ...
    {/* Right Panel */}
    <div className="w-full lg:w-1/2 ...">

// Après
<div className={`${plusJakarta.variable} font-sans min-h-screen bg-white`} data-theme="light">
  <div className="flex min-h-screen overflow-hidden">
    {/* Left Panel — contenu décoratif/marketing */}
    <aside className="hidden lg:flex lg:w-1/2 ..." aria-label="Présentation INTOWORK">
    ...
    {/* Right Panel — contenu principal */}
    <main className="w-full lg:w-1/2 ...">
```

**Test manuel :** Ouvrir VoiceOver (Mac: Cmd+F5) → utiliser le rotor (VO+U) → vérifier que "main" apparaît dans la liste des landmarks.

### Problème 2 : Messages d'erreur non liés programmatiquement aux champs

**Critère WCAG :** 3.3.1 – Identification des erreurs (Niveau A)
**Sévérité :** Critique
**Localisation :** `signin/page.tsx:47-48`, `signin/page.tsx:218-231`, `signin/page.tsx:239-252`
**Élément :** `toast.error('Email ou mot de passe incorrect')`

**Problème :**
Les erreurs de connexion sont affichées uniquement via des toasts `react-hot-toast`. Ces notifications sont visuellement transitoires et ne sont pas programmatiquement associées aux champs de formulaire concernés. Aucun `aria-invalid`, `aria-describedby`, ou `aria-live` région n'est présent.

**Impact :**
Les utilisateurs de lecteurs d'écran peuvent manquer les messages d'erreur. Les toasts de `react-hot-toast` utilisent `role="status"` par défaut (non urgent) — un toast d'erreur devrait utiliser `role="alert"` ou `aria-live="assertive"`. De plus, les champs ne signalent pas leur état d'erreur aux technologies d'assistance.

**Correction :**

```tsx
// Ajout d'un state d'erreur
const [formError, setFormError] = useState<string | null>(null);

// Dans handleSubmit — remplacer toast.error par :
if (result?.error) {
  setFormError('Email ou mot de passe incorrect');
  // Conserver le toast pour les utilisateurs voyants
  toast.error('Email ou mot de passe incorrect');
} else {
  setFormError(null);
  toast.success('Connexion réussie');
  router.push('/dashboard');
  router.refresh();
}

// Dans le JSX — ajouter une région d'erreur et aria-invalid :
{/* Zone d'erreur accessible — juste avant le formulaire */}
{formError && (
  <div
    role="alert"
    aria-live="assertive"
    className="mb-4 p-3 bg-red-50 border border-red-200 rounded-lg text-sm text-red-700"
  >
    {formError}
  </div>
)}

// Sur les inputs — ajouter aria-invalid et aria-describedby :
<input
  id="email"
  aria-invalid={formError ? 'true' : 'false'}
  aria-describedby={formError ? 'form-error' : undefined}
  ...
/>
```

**Test manuel :** Soumettre un formulaire avec des identifiants incorrects → vérifier que le lecteur d'écran annonce immédiatement l'erreur.

### Problème 3 : Icônes décoratives sans `aria-hidden="true"`

**Critère WCAG :** 1.1.1 – Contenu non textuel (Niveau A)
**Sévérité :** Sérieux
**Localisations :**

| Ligne | Composant | Contexte |
|-------|-----------|----------|
| 105 | `BuildingOfficeIcon` | Icône feature décoration |
| 119 | `UserGroupIcon` | Icône feature décoration |
| 133 | `ChartBarIcon` | Icône feature décoration |
| 146 | `ShieldCheckIcon` | Badge confiance |
| 194-199 | SVG Google | Logo dans bouton avec texte |
| 229 | `EnvelopeIcon` | Icône dans input email |
| 250 | `LockClosedIcon` | Icône dans input password |
| 288 | `ArrowRightIcon` | Icône dans bouton submit |

**Problème :**
Tous ces éléments SVG/icônes sont purement décoratifs (un label textuel adjacent existe dans chaque cas). Sans `aria-hidden="true"`, certains lecteurs d'écran tentent de les annoncer, créant du bruit inutile.

**Correction :**

```tsx
// Avant
<EnvelopeIcon className="absolute left-3 sm:left-4 top-1/2 transform -translate-y-1/2 h-4 w-4 sm:h-5 sm:w-5 text-slate-400" />

// Après — ajouter aria-hidden="true" sur chaque icône décorative
<EnvelopeIcon
  className="absolute left-3 sm:left-4 top-1/2 transform -translate-y-1/2 h-4 w-4 sm:h-5 sm:w-5 text-slate-400"
  aria-hidden="true"
/>

// Pour le SVG Google inline
<svg className="w-5 h-5" viewBox="0 0 24 24" aria-hidden="true">
  ...
</svg>

// Pour ArrowRightIcon dans le bouton
<ArrowRightIcon
  className="w-4 h-4 sm:w-5 sm:h-5 group-hover:translate-x-1 transition-transform"
  aria-hidden="true"
/>
```

### Problème 4 : Hiérarchie des titres — absence de `<h1>`

**Critère WCAG :** 1.3.1 – Info et Relations (Niveau A)
**Sévérité :** Sérieux
**Localisation :** `signin/page.tsx:96` et `signin/page.tsx:173`

**Problème :**
La page contient deux éléments `<h2>` sans `<h1>` parent :
- Ligne 96 : `<h2>La plateforme de recrutement intelligente...`
- Ligne 173 : `<h2>Connexion`

La structure du document commence à `h2`, ce qui casse la hiérarchie sémantique attendue par les lecteurs d'écran.

**Correction :**

```tsx
// Panneau gauche — contenu marketing, garder en h2 (ou l'exposer comme h1 masqué visuellement)
// Panneau droit — "Connexion" devrait être le h1 principal de la page

// Avant (ligne 173)
<h2 className="text-3xl sm:text-4xl font-bold text-slate-900 mb-2 sm:mb-3">
  Connexion
</h2>

// Après
<h1 className="text-3xl sm:text-4xl font-bold text-slate-900 mb-2 sm:mb-3">
  Connexion
</h1>

// Panneau gauche (ligne 96) — garder en h2 car c'est du contenu complémentaire
<h2 className="text-3xl xl:text-4xl font-bold mb-10 xl:mb-12 text-white leading-tight animate-fade-in animation-delay-100">
  La plateforme de recrutement intelligente pour les professionnels
</h2>
```

### Problème 5 : Titre de page non spécifique à la page de connexion

**Critère WCAG :** 2.4.2 – Titre de page (Niveau A)
**Sévérité :** Sérieux
**Localisation :** `app/layout.tsx:17` — `signin/page.tsx` (metadata absente)

**Problème :**
La page de connexion hérite du titre générique racine : `"INTOWORK Search - Plateforme de Recrutement"`. WCAG 2.4.2 exige que chaque page ait un titre décrivant son sujet ou son objectif. Un utilisateur avec plusieurs onglets ouverts ne peut pas distinguer cet onglet.

**Correction :**

```tsx
// Ajouter en haut de signin/page.tsx (avant la déclaration 'use client' si possible,
// ou dans un fichier séparé metadata.ts)

// Option recommandée — créer frontend/src/app/signin/metadata.ts
import type { Metadata } from 'next';

export const metadata: Metadata = {
  title: 'Connexion — INTOWORK Search',
  description: 'Connectez-vous à votre espace professionnel INTOWORK',
};

// Note: Les métadonnées ne peuvent pas être exportées depuis un Client Component.
// Déplacer la logique de page dans un Server Component parent si nécessaire,
// ou créer un layout.tsx spécifique à signin/.
```

### Problème 6 : État de chargement non annoncé aux lecteurs d'écran

**Critère WCAG :** 4.1.2 – Nom, Rôle, Valeur (Niveau A)
**Sévérité :** Modéré
**Localisation :** `signin/page.tsx:274-291`

**Problème :**
Le bouton de soumission affiche un spinner DaisyUI (`loading loading-spinner`) pendant le chargement. Bien que le texte du bouton change en "Connexion en cours", il n'y a ni `aria-busy` sur le formulaire, ni région `aria-live` pour annoncer le changement d'état.

**Correction :**

```tsx
// Sur le formulaire (ligne 211)
<form
  className="space-y-5 sm:space-y-6"
  onSubmit={handleSubmit}
  aria-busy={loading}
>

// Sur le bouton (ligne 274) — ajouter aria-label dynamique
<button
  type="submit"
  disabled={loading}
  aria-label={loading ? 'Connexion en cours, veuillez patienter' : 'Se connecter'}
  className="..."
>
  {loading ? (
    <>
      <span className="loading loading-spinner loading-sm" aria-hidden="true"></span>
      <span>Connexion en cours</span>
    </>
  ) : (
    <>
      <span>Se connecter</span>
      <ArrowRightIcon className="w-4 h-4 sm:w-5 sm:h-5 group-hover:translate-x-1 transition-transform" aria-hidden="true" />
    </>
  )}
</button>
```

### Problème 7 : Formulaire sans nom accessible

**Critère WCAG :** 1.3.1 – Info et Relations (Niveau A)
**Sévérité :** Modéré
**Localisation :** `signin/page.tsx:211`
**Élément :** `<form className="space-y-5 sm:space-y-6" onSubmit={handleSubmit}>`

**Problème :**
L'élément `<form>` n'a pas de nom accessible (`aria-label` ou `aria-labelledby`). Les lecteurs d'écran avancés annoncent les formulaires nommés lors de la navigation par landmarks.

**Correction :**

```tsx
// Avant
<form className="space-y-5 sm:space-y-6" onSubmit={handleSubmit}>

// Après — référencer le titre "Connexion" via aria-labelledby
// (Après avoir changé h2 en h1 avec un id)
<h1 id="signin-title" className="...">Connexion</h1>
...
<form
  className="space-y-5 sm:space-y-6"
  onSubmit={handleSubmit}
  aria-labelledby="signin-title"
>
```

### Problème 8 : Séparateur "ou" non sémantique

**Critère WCAG :** 1.3.1 – Info et Relations (Niveau A)
**Sévérité :** Mineur
**Localisation :** `signin/page.tsx:205-209`

**Problème :**
Le séparateur visuel "ou" entre la connexion Google et le formulaire n'a aucun rôle sémantique. Les lignes de séparation sont des `<div>` muets.

**Correction :**

```tsx
// Avant
<div className="flex items-center gap-3 mb-6">
  <div className="flex-1 h-px bg-slate-200"></div>
  <span className="text-xs text-slate-400 font-medium">ou</span>
  <div className="flex-1 h-px bg-slate-200"></div>
</div>

// Après
<div className="flex items-center gap-3 mb-6" role="separator" aria-label="ou">
  <div className="flex-1 h-px bg-slate-200" aria-hidden="true"></div>
  <span className="text-xs text-slate-400 font-medium" aria-hidden="true">ou</span>
  <div className="flex-1 h-px bg-slate-200" aria-hidden="true"></div>
</div>
```
